# @kit/db

Database layer for the Fastify SaaS Kit. Thin, opinionated wrapper around
**Kysely + PostgreSQL** that provides:

- a configured data source (pg Pool, CamelCase, query logging)
- a transparent transaction proxy backed by `AsyncLocalStorage`
- a generic CRUD base repository
- a migration runner + scaffolder
- typed Postgres error helpers
- a zod config fragment for `DATABASE_URL`

> If you're an AI agent: read `CLAUDE.md` for the strict conventions.
> This README is the human tour.

## What's inside

```
src/
  data-source.ts   createDataSource / closeDataSource
  transaction.ts   Trx<DB> proxy + createTransactionFactory + createTransactionStorage
  repository.ts    createBaseRepository<DB, Table>
  errors.ts        PostgresErrorCodes + isUniqueViolation / isForeignKeyViolation
  migrator.ts      migrateToLatest / rollbackLast / createMigrationFile
  config.ts        dbConfigSchema (DATABASE_URL, max connections, log flag)
```

## The big idea: `Trx<DB>`

`Trx<DB>` is a Proxy that is **both** a function and a Kysely query builder:

```ts
// Start a transaction:
await transaction(async () => {
  await usersRepository.create({ email });
  await postsRepository.create({ authorId });
});

// Or just query directly -- outside any transaction this hits the pool,
// inside a transaction scope it routes to the active Kysely Transaction:
await transaction.selectFrom('users').selectAll().execute();
```

**Why it matters:** repositories depend on `Trx<DB>`, never on a raw
`Kysely<DB>`. That means a service can wrap any group of repository calls
in `await transaction(...)` and they all join the same DB transaction
automatically -- no parameter threading, no `unitOfWork.run(...)` ceremony.
Nested `transaction(...)` calls reuse the ambient one instead of opening a
new connection, so composition across services Just Works.

The `AsyncLocalStorage` is created exactly once via
`createTransactionStorage()`. Don't import `node:async_hooks` anywhere else
or vitest's module isolation will silently break transaction propagation.

## Wiring it into your service

```ts
// services/api/src/modules/_infra/db.module.ts
import {
  createDataSource,
  createTransactionFactory,
  createTransactionStorage,
  type Trx,
} from '@kit/db';
import { asFunction, asValue } from 'awilix';
import type { Kysely } from 'kysely';
import type { DB } from '#db/schema.ts'; // generated by kysely-codegen

declare global {
  interface Dependencies {
    dataSource: Kysely<DB>;
    transactionStorage: Awaited<
      ReturnType<typeof createTransactionStorage<DB>>
    >;
    transaction: Trx<DB>;
  }
}

const dataSource = createDataSource<DB>({
  logger,
  connectionString: config.DATABASE_URL,
  maxConnections: config.DATABASE_MAX_CONNECTIONS,
  logQueries: config.DATABASE_LOG_QUERIES,
});
const transactionStorage = await createTransactionStorage<DB>();

container.register({
  dataSource: asValue(dataSource),
  transactionStorage: asValue(transactionStorage),
  transaction: asFunction(createTransactionFactory).singleton(),
});
```

After that, every `*.repository.ts` file just declares `transaction` as a
dependency:

```ts
// modules/users/users.repository.ts
import { createBaseRepository, isUniqueViolation } from '@kit/db';
import type { DB } from '#db/schema.ts';

export const createUsersRepository = ({
  transaction,
}: Pick<Dependencies, 'transaction'>) => {
  const base = createBaseRepository<DB, 'users'>(transaction, 'users');
  return {
    ...base,
    findByEmail: (email: string) =>
      transaction
        .selectFrom('users')
        .selectAll()
        .where('email', '=', email)
        .executeTakeFirst(),
  };
};

declare global {
  interface Dependencies {
    usersRepository: ReturnType<typeof createUsersRepository>;
  }
}
```

`@kit/core` auto-registers it under the key `usersRepository`. No manual
wiring.

## Base repository surface

`createBaseRepository<DB, Table>(transaction, 'tableName')` returns:

| Method                                                             | Returns                          | Notes                                             |
| ------------------------------------------------------------------ | -------------------------------- | ------------------------------------------------- |
| `findById(id)`                                                     | `T \| undefined`                 | Soft lookup                                       |
| `findByIdOrThrow(id)`                                              | `T`                              | Throws if missing                                 |
| `findAll()`                                                        | `T[]`                            | Use sparingly                                     |
| `findPaginated({ limit, offset, orderByField, orderByDirection })` | `{ data, count, limit, offset }` | Single round-trip via parallel `count` + `select` |
| `create(values)`                                                   | `T`                              |                                                   |
| `update(id, patch)`                                                | `T \| undefined`                 |                                                   |
| `deleteById(id)`                                                   | `T \| undefined`                 | Returns the deleted row                           |
| `count()`                                                          | `number`                         |                                                   |

Assumes an `id` column. Tables with composite or non-`id` primary keys
should build a bespoke repository instead of using this.

## Migrations

```ts
import {
  migrateToLatest,
  rollbackLast,
  createMigrationFile,
} from '@kit/db/migrator';

await migrateToLatest({ dataSource, migrationFolder, logger });
await rollbackLast({ dataSource, migrationFolder, logger });
await createMigrationFile(migrationFolder, 'add_users_table');
```

Wire these to `pnpm db:migrate`, `pnpm db:rollback`, and
`pnpm db:create-migration <name>` in the consuming service. Type generation
is delegated to `kysely-codegen` and lives in the service, not here.

## Error handling

`@kit/db` does **not** throw HTTP-shaped exceptions -- that's the job of
`@kit/errors`. The data layer exposes typed predicates instead:

```ts
import { isUniqueViolation, isForeignKeyViolation } from '@kit/db';

try {
  return await usersRepository.create({ email });
} catch (error) {
  if (isUniqueViolation(error)) {
    throw new ConflictException('Email already exists');
  }
  throw error;
}
```

Translate at the service layer. Keeps the data layer framework-agnostic and
easy to test.

## What this package deliberately does NOT do

- **No DI registration.** It exports factories; the service wires them.
- **No HTTP exceptions.** See `@kit/errors`.
- **No schema generation.** Use `kysely-codegen` in the consuming service.
- **No multi-database support.** PostgreSQL only, by design.
